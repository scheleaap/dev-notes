# Requirements for drafting bug reports

Bug reports should be structured as follows. Everything prefixed with `>` or surrounded with `<` and `>` is guidance for the author and should not be part of the actual issue.

```
# <Bug report title>

> Summary of the bug

## How to reproduce

> Minimal steps to reproduce the issue

## Expected behavior

> When running the steps to reproduce, what is supposed to happen?

## Actual behavior

> When running the steps to reproduce, what actually happens?

## Hints

> Include this section only if there is any relevant information.
> It can include, but is not limited to:
> * Any additional information that could be relevant
> * Hypotheses on what the cause of the problem
> * ...
```

You must distinguish between what you know for a fact and speculation/conjecture. The focus of the bug report must lie on what happens. Facts go into the first sections, speculation goes into the last section.
